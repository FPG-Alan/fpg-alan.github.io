从输入地址到获取页面详细过程

2017-03-23

之前看过一到面试题,需要尽可能的详细描述用户在浏览器中打开一个网页的过程,目前我把《HTTP权威指南》看到了第五章,在这里试着描述一下。

这个过程可以说的很简单,也可以很复杂,我认为主要还是

  1. 对HTTP乃至TCP/IP的理解
  2. 对浏览器如何解析HTML、CSS、JS等资源的理解。
    所以今天的描述只代表我目前的理解程度。
  1. 用户打开浏览器,在地址栏输入目标地址,按下回车或点击按钮。
  2. 浏览器接收目标地址,从目标地址(URL)中解析出方案(协议),端口号(默认80),以及可能会存在的查询条件。
  3. 浏览器检查缓存表中是否存在对应IP,如果存在则直接获取IP,如果不存在则连接DNS域名解析服务器,获取对应IP。
  4. 浏览器根据上一步获得的IP地址,开始尝试连接目标主机
    4.1 向目标主机发送一个40-80字节的TCP分组,带有一个SYN标志,用来说明这是一个连接请求
    4.2 如果一切顺利,目标服务器对一些连接参数进行计算,并返回一个TCP分组,这个分组中,存在SYN和ACK标志位,说明连接请求已经被接受
    4.3 浏览器向目标服务器发送一个TCP分组通知它连接已经建立
  5. TCP连接已经被建立,浏览器根据第二步中解析出的地址向目标服务器发送一个方法为 GET 的HTTP事务请求报文。
  6. 目标服务器接受请求报文, 解析请求行, 并将解析出的报文缓存在内存中, 直到收到全部请求报文并足以解析理解其意义为止。
  7. 如果一切顺利, 目标服务器解析请求报文完成, 将其中的URI映射到本地的对应地址, 并开始读取对应文件内容。
  8. 目标服务器开始构建响应报文, 包含响应状态码, 响应首部, 以及响应主体(在这个分析案例中,相应主体应该存在并且为对应站点首页)
  9. 浏览器收到响应报文, 并开始解析报文主体(这里已Chrome为例)
    9.1 呈现引擎(下文称为webkit)读取文档内容
    9.2 解析HTML文档, 转化各个节点, 构建DOM树结构
    9.3 解析内联CSS样式数据, 构建渲染树, 渲染树会包含多个带有视觉属性(如颜色和尺寸)的矩形。这些矩形的排列顺序就是它们将在屏幕上显示的顺序。
    9.4 对渲染树进行布局处理, 为每个节点分配一个应出现在屏幕上的确切坐标。
    9.5 webkit 遍历渲染树, 由浏览器的用户界面后端层将每个节点绘制出来。
    注意, 9.2-9.5是一个渐进式的过程, 为了提高用户体验, webkit并不会等待所有html内容都被解析完成后才进行绘制,而是解析一点绘制一点。
  10. 如果当前页面存在其他请求(外部CSS文件、JS文件), 浏览器会对这些文件重复步骤5-9。(这些请求一般是同步发起的)。

global, cmd/amd, ES6 Module Js的模块化进程

2017-03-03
technology stack

为什么模块化对js很重要:

  • 模块化鼓励开发者使用小的低耦合模块，提供清晰的接口，而不是大块的代码.
  • 有助于测试，因为模块可以被实现了相同接口的模拟模块替换，即使在运行时。
  • 提升了代码的可维护性，因为小的低耦合模块更易懂。

Global
早期的前端开发,我们使用script标签引入JS,一个文件作为一个模块,模块接口被直接暴露在全局作用域下,也就是window对象。各个模块可以通过这个全局对象访问依赖,这种方式会造成全局作用域的变量名污染,多个script标签的顺序会影响每个模块(文件)的加载顺序,开发者需要自己解决模块对库或其他模块的依赖引入问题,项目越大,也就越难管理,明显不是一种好的解决方案。

CMD/AMD
CMD即 Commonjs规范, 一个模块可以使用exports定义自己的导出, 使用require方法用于加载依赖, 这个方法是同步的。
这种规范主要被nodejs所实现, 他的好处在于可以更好的组织模块, 重用代码, 并且在nodejs的依赖库(npm)里, 已经存在很多可以直接使用的模块, 但是这个规范使用同步的方式加载依赖, 适用于服务端, 在客户端则不适用。

AMD即 异步模块定义, 主要被requirejs实现, 相对于Commonjs, 是专门为客户端提出的一种模块化定义, 提供了加载和定义模块的规范, 同时可以异步加载模块, 可以同步加载多个模块。这也是我早期主要实用的模块定义方案。

ES6模块
是ECMAScript 2015提出的, js原生的模块化方案, 这个方案目前浏览器支持还不太好, 需要通过Babel进行转译。

到了ES6模块这里,很多通用模块加载器出现了,比如Angularjs2.0使用的Systemjs以及Webpack, 他们都可以加载上面的所有模块定义。

源于看到了一段不是很懂得代码, 使用了按位或运算符。

平时在项目中很少会用到,最多用移位运算符进行加减运算,这次既然看到了就整理整理,在适合的地方用位运算符可以提高一些性能(显得很酷…)。

位运算符包括 &, |, ^, ~, <<, >>, >>>, 分别是按位与, 按位或, 按位异或, 按位非, 左移, 有符号右移, 无符号右移

需要注意的是几个移位运算符。
左移会将数值的所有位向左移动指定的位数。在左移后,原数值右侧空出的位由0填补。
左移一位其实就相当于将原数值乘以2,不会影响操作数的符号位。

有符号右移会将数值向右移动,但保留符号位。在移位过程中,空缺位出现在原数值的左侧,符号位的右侧,用符号位的值来填充空位。
右移一位相当于原数除2后向下取整。

无符号右移会将数值的所有32位都向右移动。对正数来说,无符号右移的结果和有符号相同,但是对负数来说就不一样了,无符号右移会把负数的符号位也进行移动,左边空出位置用0填充。

一.反射(Reflect) 代理(Proxy)

Reflect 和 Proxy 都是ES6新加入的操作对象的API,两者往往是配合使用。
Reflect 是一个静态对象,类似Math,主要提供了几个原本位于Object上的方法, 比如 defineProperty, assign, apply, 并对这些方法都做了优化, 例如Object.defineProperty(obj, name, desc)在无法定义属性时，会抛出一个错误，而Reflect.defineProperty(obj, name, desc)则会返回false。

此外,某些Object操作是命令式，比如name in obj和delete obj[name]，而Reflect.has(obj, name)和Reflect.deleteProperty(obj, name)让它们变成了函数行为。

最后, Reflect 可以和 Proxy配合使用, Reflect对象的方法与Proxy对象的方法一一对应，只要是Proxy对象的方法，就能在Reflect对象上找到对应的方法。这就让Proxy对象可以方便地调用对应的Reflect方法，完成默认行为，作为修改行为的基础。也就是说，不管Proxy怎么修改默认行为，你总可以在Reflect上获取默认行为。

这里就要说到 Proxy , Proxy 用于修改某些操作的默认行为，等同于在语言层面做出修改，所以属于一种“元编程”（meta programming），即对编程语言进行编程。

三.异步编程 Promise, Generator, async

Promise 是异步编程的一种解决方案，比传统的解决方案——回调函数和事件——更合理和更强大。它由社区最早提出和实现，ES6将其写进了语言标准，统一了用法，原生提供了Promise对象。
所谓Promise，简单说就是一个容器，里面保存着某个未来才会结束的事件（通常是一个异步操作）的结果。从语法上说，Promise 是一个对象，从它可以获取异步操作的消息。Promise 提供统一的 API，各种异步操作都可以用同样的方法进行处理。
Promise对象有以下两个特点。

  • 对象的状态不受外界影响。Promise对象代表一个异步操作，有三种状态：Pending（进行中）、Resolved（已完成，又称 Fulfilled）和Rejected（已失败）。只有异步操作的结果，可以决定当前是哪一种状态，任何其他操作都无法改变这个状态。这也是Promise这个名字的由来，它的英语意思就是“承诺”，表示其他手段无法改变。

  • 一旦状态改变，就不会再变，任何时候都可以得到这个结果。Promise对象的状态改变，只有两种可能：从Pending变为Resolved和从Pending变为Rejected。只要这两种情况发生，状态就凝固了，不会再变了，会一直保持这个结果。就算改变已经发生了，你再对Promise对象添加回调函数，也会立即得到这个结果。这与事件（Event）完全不同，事件的特点是，如果你错过了它，再去监听，是得不到结果的。

有了Promise对象，就可以将异步操作以同步操作的流程表达出来，避免了层层嵌套的回调函数。此外，Promise对象提供统一的接口，使得控制异步操作更加容易。

Promise也有一些缺点。首先，无法取消Promise，一旦新建它就会立即执行，无法中途取消。其次，如果不设置回调函数，Promise内部抛出的错误，不会反应到外部。第三，当处于Pending状态时，无法得知目前进展到哪一个阶段（刚刚开始还是即将完成）。

如果某些事件不断地反复发生，一般来说，使用 stream 模式是比部署Promise更好的选择。

Generator 函数是 ES6 提供的一种异步编程解决方案，语法行为与传统函数完全不同。
Generator 函数有多种理解角度。从语法上，首先可以把它理解成，Generator 函数是一个状态机，封装了多个内部状态。
执行 Generator 函数会返回一个遍历器对象，也就是说，Generator 函数除了状态机，还是一个遍历器对象生成函数。返回的遍历器对象，可以依次遍历 Generator 函数内部的每一个状态。
形式上，Generator 函数是一个普通函数，但是有两个特征。一是，function关键字与函数名之间有一个星号；二是，函数体内部使用yield语句，定义不同的内部状态（yield在英语里的意思就是“产出”）。

上面代码定义了一个Generator函数helloWorldGenerator，它内部有两个yield语句“hello”和“world”，即该函数有三个状态：hello，world和return语句（结束执行）。

然后，Generator函数的调用方法与普通函数一样，也是在函数名后面加上一对圆括号。不同的是，调用Generator函数后，该函数并不执行，返回的也不是函数运行结果，而是一个指向内部状态的指针对象，也就是上一章介绍的遍历器对象（Iterator Object）。

下一步，必须调用遍历器对象的next方法，使得指针移向下一个状态。也就是说，每次调用next方法，内部指针就从函数头部或上一次停下来的地方开始执行，直到遇到下一个yield语句（或return语句）为止。换言之，Generator函数是分段执行的，yield语句是暂停执行的标记，而next方法可以恢复执行。


上面代码一共调用了四次next方法。

第一次调用，Generator函数开始执行，直到遇到第一个yield语句为止。next方法返回一个对象，它的value属性就是当前yield语句的值hello，done属性的值false，表示遍历还没有结束。

第二次调用，Generator函数从上次yield语句停下的地方，一直执行到下一个yield语句。next方法返回的对象的value属性就是当前yield语句的值world，done属性的值false，表示遍历还没有结束。

第三次调用，Generator函数从上次yield语句停下的地方，一直执行到return语句（如果没有return语句，就执行到函数结束）。next方法返回的对象的value属性，就是紧跟在return语句后面的表达式的值（如果没有return语句，则value属性的值为undefined），done属性的值true，表示遍历已经结束。

第四次调用，此时Generator函数已经运行完毕，next方法返回对象的value属性为undefined，done属性为true。以后再调用next方法，返回的都是这个值。

总结一下，调用Generator函数，返回一个遍历器对象，代表Generator函数的内部指针。以后，每次调用遍历器对象的next方法，就会返回一个有着value和done两个属性的对象。value属性表示当前的内部状态的值，是yield语句后面那个表达式的值；done属性是一个布尔值，表示是否遍历结束。

ES2017 标准引入了 async 函数，使得异步操作变得更加方便。
async 函数是什么？一句话，它就是 Generator 函数的语法糖。
async函数就是将 Generator 函数的星号（*）替换成async，将yield替换成await，仅此而已。

async函数对 Generator 函数的改进，体现在以下四点。

  • 内置执行器。
    Generator 函数的执行必须靠执行器，所以才有了co模块，而async函数自带执行器。也就是说，async函数的执行，与普通函数一模一样，只要一行。
    var result = asyncReadFile();
    上面的代码调用了asyncReadFile函数，然后它就会自动执行，输出最后结果。这完全不像 Generator 函数，需要调用next方法，或者用co模块，才能真正执行，得到最后结果。

  • 更好的语义。
    async和await，比起星号和yield，语义更清楚了。async表示函数里有异步操作，await表示紧跟在后面的表达式需要等待结果。

  • 更广的适用性。
    co模块约定，yield命令后面只能是 Thunk 函数或 Promise 对象，而async函数的await命令后面，可以是Promise 对象和原始类型的值（数值、字符串和布尔值，但这时等同于同步操作）。

  • 返回值是 Promise。
    async函数的返回值是 Promise 对象，这比 Generator 函数的返回值是 Iterator 对象方便多了。你可以用then方法指定下一步的操作。

进一步说，async函数完全可以看作多个异步操作，包装成的一个 Promise 对象，而await命令就是内部then命令的语法糖。

nodejs 版本管理

2017-02-28
javascript nodejs

使用 Node 的过程中会经常遇到需要特定的 Node 版本的情况,一个 Node 版本管理工具是必备的。
目前比较流行的版本管理有两个,分别是 nvm 和 n ,这两个版本管理我都有用过。

一开始是用 nvm,这个东西独立于 Node,是一个独立的安装包, 在安装的时候, nvm 将不同的 Node 版本存储到 ~/.nvm/<version>/ 下，然后修改 $PATH，将指定版本的 Node 路径加入,这样我们调用的 Node 命令即是使用指定版本的 Node。

n则更为简单一些,它依赖 Node, 使用 npm 安装, n 会将指定版本的 Node 保存下来, 然后在我们切换版本的时候,将对应版本的 Node 文件复制到 usr/local/bin 目录下从而切换 Node 版本。

两者的原理弄清楚了也就好比较两者的区别:
n 是个比较简单易用的工具, 但安装 n 之前你需要先安装 Node 环境, 并且 n 这种简单复制的做法无法处理全局安装的 Node 模块, 一旦切换了版本, 全局包就有可能执行出错。
nvm 则不存在全局包的问题, 在 nvm 下全局安装的包也会被安装到 ~/.nvm/<version>/ 下, 切换版本之后就需要重新安装全局包。 但是因为 nvm 会改写 $PATH, 所以有些IDE或者其他程序(比如 lessc )可能会需要给定一个完整的 Node 路径。

目前我使用的是 n ,因为我的工作中较少出现需要频繁切换 Node 环境的场景, 使用版本管理更多的是便于更新到最新的 Node 环境, 如果有需要在不同 Node 环境之间来回切换, 还是倾向于选择 nvm。
